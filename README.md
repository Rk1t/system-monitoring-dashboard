# Система визуализации мониторинга данных компьютерных систем

## 1. Назначение проекта

Проект разрабатывается как ВКР на тему: **«Разработка модуля визуализации мониторинга данных компьютерных систем»**.

На раннем этапе система работала как локальный мониторинг одного ПК: backend собирал метрики текущего компьютера через `psutil`, сохранял их в SQLite и показывал dashboard во Vue-интерфейсе.

Текущая архитектура постепенно развивается в сторону **централизованной системы мониторинга**:

- центральный backend принимает, хранит и анализирует телеметрию;
- web-интерфейс администратора показывает контролируемые системы;
- каждая система представляется как отдельный узел `Node`;
- метрики разных систем хранятся раздельно;
- в дальнейшем обычные Windows/Linux машины смогут отправлять данные через Monitoring Agent;
- серверное оборудование в перспективе сможет подключаться через HPE iLO / Dell iDRAC по Redfish API.

На текущем этапе агент и Redfish еще не вынесены в отдельные компоненты. Локальный сбор через `psutil` сохранен для обратной совместимости и разработки.

## 2. Текущая архитектура

Текущие компоненты проекта:

- **Central Backend FastAPI**  
  Центральный backend-сервер. Сейчас он одновременно выполняет роль локального сборщика метрик для `Local System`, но структура уже подготовлена к нескольким узлам.

- **Vue Web UI**  
  Web-интерфейс администратора. Пока основной dashboard отображает локальный узел, но backend уже предоставляет node-aware API для будущей страницы списка систем.

- **SQLite / SQLAlchemy**  
  SQLite используется как простое локальное хранилище. SQLAlchemy описывает модели `Node` и `Metric`.

- **Local System node**  
  Специальный узел с `source_type="local"`. Он создается автоматически при запуске backend и представляет компьютер, на котором сейчас запущен backend.

- **Будущий Monitoring Agent**  
  В следующем этапе текущий `psutil`-сборщик можно вынести в легкий агент, который будет запускаться на Windows/Linux системах и отправлять метрики на центральный backend.

- **Будущие источники HPE iLO / Dell iDRAC через Redfish**  
  В перспективе можно добавить источники серверной телеметрии: состояние питания, температуры, вентиляторов, аппаратные ошибки и другие данные, доступные через Redfish API.

Общая целевая схема:

```text
Monitoring Agent / Redfish Source
          |
          v
Central FastAPI Backend
          |
          v
SQLite / SQLAlchemy
          |
          v
Vue Web UI
```

## 3. Почему появилась модель узлов

Изначально все записи в таблице `metrics` относились к одному локальному компьютеру. Для централизованного мониторинга этого недостаточно: система должна понимать, от какого компьютера или сервера пришла конкретная телеметрия.

Поэтому добавлена сущность **Node**.

`Node` описывает контролируемую систему:

- локальный компьютер;
- будущий агент на Windows/Linux;
- будущий источник HPE iLO;
- будущий источник Dell iDRAC.

Теперь каждая запись метрик содержит `node_id`, то есть привязку к конкретному узлу. Это позволяет:

- хранить историю каждого узла отдельно;
- строить графики конкретной системы;
- выполнять анализ состояния конкретного узла;
- позже добавить список систем во frontend;
- расширить проект без переписывания всей базы.

## 4. Модель данных

### Таблица `nodes`

Таблица хранит список контролируемых систем.

Основные поля:

- `id`  
  Уникальный идентификатор узла.

- `name`  
  Отображаемое имя системы. Для локального режима используется `Local System`.

- `hostname`  
  Имя компьютера или сервера.

- `source_type`  
  Тип источника телеметрии. Сейчас используется `local`. Архитектурно предусмотрены значения:
  - `local`;
  - `agent`;
  - `ilo`;
  - `idrac`.

- `os_name`  
  Название и версия операционной системы или платформы.

- `agent_version`  
  Версия агента. Для локального режима может быть пустым.

- `status`  
  Текущий статус узла. Сейчас для локального узла устанавливается `online`.

- `health_score`  
  Общий индекс состояния системы от 0 до 100. Значение обновляется при анализе метрик.

- `first_seen`  
  Дата и время первого появления узла.

- `last_seen`  
  Дата и время последнего обновления узла или получения метрик.

### Таблица `metrics`

Таблица хранит историю основных метрик.

Основные поля:

- `id`  
  Уникальный идентификатор записи.

- `node_id`  
  Ссылка на `nodes.id`. Позволяет хранить метрики разных систем отдельно.

- `timestamp`  
  Время сбора метрики.

- `cpu_percent`  
  Процент загрузки CPU.

- `ram_percent`  
  Процент использования оперативной памяти.

- `disk_percent`  
  Процент использования основного диска.

- `bytes_sent`  
  Общее количество отправленных байт.

- `bytes_recv`  
  Общее количество полученных байт.

- `network_sent_per_sec`  
  Скорость отправки данных в байтах в секунду.

- `network_recv_per_sec`  
  Скорость получения данных в байтах в секунду.

Важно: подробные данные о CPU cores, дисках, сетевых интерфейсах и процессах пока не сохраняются в SQLite. Они считываются через `psutil` по запросу.

## 5. Режим обратной совместимости

Старые endpoints сохранены и продолжают работать:

- `GET /api/metrics/current`
- `GET /api/metrics/history`
- `GET /api/metrics/summary`
- `GET /api/analysis/summary`

Теперь внутри backend они работают через узел `Local System`.

Это значит:

- текущий frontend не ломается;
- локальный режим разработки сохраняется;
- существующая логика анализа продолжает работать;
- старые метрики из SQLite автоматически привязываются к локальному узлу.

## 6. Логика Local System

При запуске backend выполняет инициализацию:

1. Создает таблицы SQLAlchemy, если их еще нет.
2. Проверяет старую SQLite-базу.
3. Если в таблице `metrics` нет поля `node_id`, добавляет его безопасной миграцией.
4. Создает локальный узел `Local System`, если его еще нет.
5. Заполняет:
   - `hostname` через `socket.gethostname()`;
   - `os_name` через `platform.system()` и `platform.platform()`;
   - `source_type="local"`;
   - `status="online"`;
   - `first_seen` и `last_seen`.
6. Старые записи метрик без `node_id` привязывает к `Local System`.
7. Новые локальные метрики сохраняет уже с `node_id` локального узла.

Таким образом проект не падает при запуске на старой базе данных.

## 7. API

Основной адрес при запуске backend:

```text
http://127.0.0.1:8000
```

Документация FastAPI:

```text
http://127.0.0.1:8000/docs
```

### Старые endpoints

- `GET /api/metrics/current`  
  Собирает текущие локальные метрики и сохраняет их для `Local System`.

- `GET /api/metrics/history?limit=N`  
  Возвращает историю метрик локального узла.

- `GET /api/metrics/summary?limit=N`  
  Возвращает средние и максимальные значения CPU, RAM и Disk по локальному узлу.

- `GET /api/analysis/summary?limit=N`  
  Возвращает анализ состояния локального узла: тренды, health score, аномалии, узкое место и рекомендуемый интервал обновления.

### Новые node-aware endpoints

- `GET /api/nodes`  
  Возвращает список контролируемых узлов. Сейчас минимум один узел: `Local System`.

- `GET /api/nodes/{node_id}`  
  Возвращает подробную информацию об узле: имя, hostname, тип источника, ОС, статус, health score, first_seen и last_seen.

- `GET /api/nodes/{node_id}/metrics/current`  
  Возвращает текущую или последнюю метрику конкретного узла. Для узла `source_type="local"` дополнительно выполняется локальный сбор через `psutil`.

- `GET /api/nodes/{node_id}/metrics/history?limit=N`  
  Возвращает историю метрик конкретного узла.

- `GET /api/nodes/{node_id}/metrics/summary?limit=N`  
  Возвращает summary по конкретному узлу.

- `GET /api/nodes/{node_id}/analysis/summary?limit=N`  
  Выполняет анализ состояния конкретного узла на центральном backend.

## 8. Аналитика и визуализация

Backend выполняет простую объяснимую аналитику:

- `Trend Analysis`  
  Сравнивает первую и последнюю точку выбранного окна истории.

- `Health Score`  
  Рассчитывает общий индекс состояния от 0 до 100.

- `Anomaly Detection`  
  Сравнивает текущее значение со средним по последним N точкам.

- `Bottleneck Detection`  
  Определяет предполагаемое узкое место: CPU, RAM, Disk или Network.

- `Adaptive Refresh Rate`  
  Возвращает рекомендуемый интервал обновления интерфейса.

Во frontend уже есть:

- карточки CPU, RAM, Disk, Network;
- графики CPU и RAM;
- downsampling графиков;
- режим сравнения полного и оптимизированного отображения;
- предупреждения;
- блок health score;
- блок аномалий;
- блок предполагаемого узкого места.

## 12. Направление дальнейшего развития

Следующие безопасные этапы развития:

1. **Вынос psutil-сборщика в Monitoring Agent**  
   Текущий локальный сборщик можно перенести в отдельную папку `agent/`.

2. **Добавление agent register / telemetry endpoints**  
   Backend должен принимать регистрацию агента и метрики от внешних систем:
   - `POST /api/agent/register`;
   - `POST /api/agent/{node_id}/metrics`;
   - `POST /api/agent/{node_id}/heartbeat`.

3. **Страница списка систем во frontend**  
   Главная страница должна показывать список узлов, статус, hostname, source type, last_seen и health score.

4. **Страница конкретного узла**  
   Текущий dashboard должен стать страницей выбранного узла.

5. **Поддержка HPE iLO / Dell iDRAC через Redfish**  
   В перспективе можно добавить источники аппаратной телеметрии серверов.

6. **Автоматизированная диагностика состояния узлов**  
   Анализ должен выполняться на центральном backend для каждого узла отдельно.

Важно: проект не использует JWT, Docker, PostgreSQL, WebSocket, микросервисы, машинное обучение или тяжелые аналитические библиотеки. Архитектура развивается постепенно и остается подходящей для объяснения на защите ВКР.
